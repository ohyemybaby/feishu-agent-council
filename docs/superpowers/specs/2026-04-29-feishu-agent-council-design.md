# Feishu Agent Council MVP Design

## 1. Goal

Build a Feishu-based multi-agent assistant that lets the user ask one question in a Feishu chat and receive a concise, structured, high-value answer generated by multiple models.

The project will be built in three phases:

1. Brainstorming and decision analysis.
2. Computer and code diagnostics.
3. Data analysis and forecasting.

The first MVP focuses only on phase 1: brainstorming and decision analysis. It will use GLM and DeepSeek first, then leave extension points for Claude, Codex, local tools, data tools, and additional providers.

## 2. First MVP Scope

### In Scope

- Receive user messages from a Feishu bot in a group chat.
- Detect the requested mode: quick, standard, deep, or research.
- Send the prompt to GLM and DeepSeek as independent analysts.
- Ask a host agent to summarize, compare, and compress their views.
- Return a structured Feishu message with:
  - one-sentence conclusion
  - best 3-5 ideas
  - feasibility score
  - risks
  - assumptions
  - next actions
- Track basic token/cost estimates per request.
- Store task records and model outputs for later quality review.

### Out of Scope for MVP

- Direct computer control.
- Running local shell commands from Feishu.
- Reading private files automatically.
- Web scraping or external data collection.
- Forecasting from spreadsheets.
- Fully autonomous long-running tasks.
- Claude Code, Codex, Trae, or local IDE control.

These are intentionally deferred so the first version can validate the core interaction: Feishu input, multi-model analysis, debate-style synthesis, readable output, and budget control.

## 3. User Experience

The user interacts from Feishu, including mobile.

Example input:

```text
@AI-Council /deep Analyze whether a Feishu multi-agent decision assistant is worth building.
```

Example output:

```text
Conclusion:
Worth building, but the first version should focus on decision brainstorming and readable synthesis instead of autonomous computer control.

Best Ideas:
1. Multi-model value comes from role separation, not model count.
2. Feishu is a strong entry point because mobile and group workflows are natural.
3. The first version should enforce strict permission boundaries.

Feasibility:
Technical feasibility: high
Cost pressure: medium
Safety risk: medium
Recommended priority: A-

Main Risks:
1. Long outputs may reduce usefulness.
2. Multi-model calls may waste tokens.
3. Autonomous execution creates permission risk if added too early.

Next Actions:
1. Connect the Feishu bot.
2. Connect GLM and DeepSeek.
3. Implement standard and deep modes.
4. Add structured synthesis.
```

Detailed raw model outputs should not be sent by default. They can be stored and optionally exposed later through a Feishu document, debug command, or admin panel.

## 4. Modes

### Quick Mode

- One model answers.
- Host compresses the result.
- Lowest token use.
- Used for simple questions.

### Standard Mode

- GLM and DeepSeek answer independently.
- Host merges and compresses.
- Default mode.

### Deep Mode

- GLM and DeepSeek answer independently.
- Each model critiques the other's extracted ideas.
- Host produces a final synthesis.
- Reviewer checks risks, assumptions, and readability.

### Research Mode

Reserved for later. It will add web search, file parsing, Feishu Sheets, data tools, and citations.

## 5. Multi-Agent Debate Flow

The system should avoid free-form group chat between models. It should use a controlled debate workflow.

1. Host rewrites the user's question into a clear task.
2. GLM and DeepSeek answer independently.
3. Host extracts each answer into structured claims.
4. Critic step identifies weak assumptions, missing evidence, and implementation risks.
5. Host asks for revised conclusions if deep mode is enabled.
6. Final synthesizer produces the user-facing response.
7. Reviewer checks:
   - whether the answer is readable
   - whether risks are visible
   - whether claims are overconfident
   - whether the next actions are practical

The final Feishu response should be short by default. The goal is not to show every model's full thinking; the goal is to give the user a useful decision artifact.

## 6. Agent Roles

### Host

- Clarifies the question.
- Chooses the mode.
- Routes the task to model providers.
- Extracts key claims.
- Produces final synthesis.

### Strategy Analyst

- Looks at opportunity, value, market logic, and trade-offs.
- Can be backed by GLM or DeepSeek in MVP.

### Engineering Analyst

- Looks at implementation cost, technical risks, dependencies, and maintainability.
- Can be backed by DeepSeek in MVP.

### Skeptic

- Finds flaws, missing evidence, hidden cost, safety issues, and unrealistic assumptions.
- Can be implemented as a prompt role using either model.

### Reviewer

- Checks final output quality, confidence, and readability.
- Can be implemented as a low-token review prompt.

## 7. Token Budget Strategy

Default behavior should minimize token use.

- Standard mode is the default.
- Deep mode requires explicit wording, such as "/deep", "deep analysis", or "debate".
- Quick mode can be triggered by "/quick" or short/simple questions.
- Each provider response should be constrained by schema and length.
- Long histories should be summarized instead of sent in full.
- Raw outputs should be stored but not resent to every model.
- The host should pass extracted claims to the critic, not the entire original answer when possible.
- Research mode should show estimated cost before doing expensive work.

## 8. Safety and Permission Boundary

MVP does not execute local commands or modify files.

Later phase B may support computer and code diagnostics with strict permission levels:

- Safe automatic actions:
  - read logs
  - inspect project structure
  - check package versions
  - check port usage
  - analyze error messages
- Confirmation-required actions:
  - install dependencies
  - modify files
  - delete files
  - change system settings
  - run unknown scripts
  - commit or push code

Feishu mobile can request these actions later, but high-risk execution must require an explicit confirmation button.

## 9. Accuracy Evaluation

The system should evaluate model quality by task type over time.

Suggested scoring dimensions:

- correctness
- feasibility
- evidence quality
- risk awareness
- readability
- novelty
- final task success

Each completed task should store:

- user prompt
- task type
- selected mode
- participating providers
- final answer
- model self-confidence
- user feedback if provided
- token/cost estimate
- whether the answer solved the problem

This allows future routing decisions such as using DeepSeek more for engineering feasibility, GLM more for Chinese context, and Claude/Codex later for higher-end synthesis or code execution.

## 10. Data Model

Minimum tables or collections:

- tasks
  - id
  - feishu_chat_id
  - feishu_message_id
  - user_id
  - task_type
  - mode
  - status
  - created_at
  - completed_at
- model_runs
  - id
  - task_id
  - provider
  - role
  - prompt_tokens
  - completion_tokens
  - cost_estimate
  - latency_ms
  - output_json
- final_answers
  - id
  - task_id
  - answer_text
  - answer_json
  - quality_score
- feedback
  - id
  - task_id
  - rating
  - comment
  - created_at

SQLite is enough for the local MVP. Postgres can be introduced later if the system becomes multi-user or production-facing.

## 11. Architecture

Recommended MVP architecture:

```text
Feishu Group Chat
  -> Feishu Bot Webhook
  -> Backend API
  -> Task Router
  -> Orchestrator
  -> Provider Adapters
       -> GLM
       -> DeepSeek
  -> Synthesis Pipeline
  -> Storage
  -> Feishu Reply
```

Suggested implementation stack:

- Backend: Python FastAPI or Node.js.
- Storage: SQLite for MVP.
- Queue: in-process queue first; Redis later if needed.
- Providers: isolated adapter modules.
- Config: environment variables for API keys and Feishu credentials.

Python FastAPI is recommended because later phase C will need data analysis and forecasting tools.

## 12. Provider Adapter Contract

Each model provider should expose the same interface:

```text
run_model(
  provider,
  role,
  system_prompt,
  user_prompt,
  response_schema,
  max_tokens,
  temperature
) -> ModelRunResult
```

The rest of the system should not depend on provider-specific request formats.

## 13. Feishu Commands

Initial commands:

```text
/quick question
/standard question
/deep question
/save-token question
/help
```

Natural language triggers should also work:

- "simple answer"
- "deep analysis"
- "debate this"
- "feasibility analysis"
- "brainstorm this"

Chinese command aliases should be supported in the product, but the spec keeps command examples ASCII to avoid encoding ambiguity in local tooling.

## 14. Phase Roadmap

### Phase A: Brainstorming and Decision Analysis

Goal: produce high-quality structured advice from multiple models in Feishu.

Milestones:

1. Feishu bot receives and replies to messages.
2. GLM adapter works.
3. DeepSeek adapter works.
4. Standard mode works.
5. Deep mode works.
6. Token/cost tracking works.
7. Basic feedback command works.

### Phase B: Computer and Code Diagnostics

Goal: diagnose project and computer problems with controlled permissions.

Milestones:

1. Local agent service connects to the orchestrator.
2. Read-only diagnostics are supported.
3. Risk classifier identifies unsafe actions.
4. Feishu confirmation buttons are added.
5. Code/project diagnostics are summarized in readable format.

### Phase C: Data Analysis and Forecasting

Goal: analyze files, Feishu Sheets, and historical data.

Milestones:

1. CSV and Excel ingestion.
2. Feishu Sheets integration.
3. Data cleaning and profiling.
4. Charts and summary statistics.
5. Forecasting tools.
6. Business-readable report generation.

## 15. Open Decisions

These should be confirmed before implementation:

1. Backend stack: Python FastAPI is recommended.
2. Hosting: local machine first or cloud server first.
3. Feishu app type: internal app is recommended.
4. Provider API availability: confirm GLM and DeepSeek API keys.
5. Reply format: plain text first or Feishu interactive cards first.

## 16. Acceptance Criteria for MVP

MVP is successful when:

1. The user can ask a question from Feishu mobile or desktop.
2. The backend receives the message.
3. GLM and DeepSeek both produce independent structured analysis.
4. The host creates a short final answer.
5. The Feishu bot replies in the same chat.
6. The answer is readable in under one minute.
7. The system records provider, mode, token estimate, and final answer.
8. The user can choose quick, standard, or deep mode.
